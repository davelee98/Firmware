01

Power & sleep lifecycle

4 functions
pollActivity() main.cpp:144 ESP32

Runs first on every ESP32 pass. Compares BLE queue heads, connection count, LAN-session state, the dynamic advertising payload, and the bleRestartAdvertisingPending flag against their previous values; if anything changed — or work is simply in flight — it stamps lastActivityMs = millis(). That timestamp is the sole input to the deep-sleep quiet window, so this function is what keeps the device awake while it's busy.

Edge-detects connect+drop blips that leave connCount at 0 on both sides.

enterDeepSleep(bool force = false) main.cpp:399 ESP32

Puts the ESP32 into timer-wake deep sleep. Bails early if not battery-powered (power_mode != 1), if deep_sleep_time_seconds == 0, or — unless force — if a BLE client is connected (re-checked here to avoid tearing down a live link). Otherwise sets woke_from_deep_sleep, stops advertising, deinits the BLE stack, arms the wake timer, latches power for sleep, and calls esp_deep_sleep_start() — which does not return.

fullSetupAfterConnection() main.cpp:379 ESP32

Called once when a client connects during the post-wake advertising window. Promotes the device from the lightweight wake state to full operation: brings up WiFi (initWiFi(false)) and, for bb_epaper panels, zeroes and re-initializes the panel driver (bbepSetPanelType / rotation). Seeed-GFX panels skip the bb_epaper re-init.

idleDelay(uint32_t delayMs) main.cpp:361 Both

An interruptible replacement for delay(). Sleeps in 100 ms chunks, and between chunks keeps the device responsive: nRF advertising tick, button events, touch input, and LED flash all get serviced. Used everywhere the loop wants to pause without going deaf to input.

02

BLE & command path

5 functions
imageDataWritten(conn, chr, data, len) communication.cpp:482 Both

The central command dispatcher. Reads a 2-byte big-endian opcode and routes to the matching handler (authenticate, firmware version, image write, etc.), enforcing the authentication/encryption gate for everything except 0x0050 and 0x0043. In the loop it's driven from the ESP32 command queue; it's also the BLE write callback on nRF and the LAN frame handler on WiFi — the single funnel all commands pass through.

esp32_ble_notify_enabled() ble_init.cpp:153 ESP32

Predicate: true only when a client is connected and has subscribed to notifications (CCCD 0x2902 enabled, or the NimBLE subscribe flag). The loop uses it to decide whether the response queue can actually be flushed — if a client is connected but hasn't enabled notifications yet, responses stay queued rather than being dropped.

esp32_restart_ble_advertising() ble_init.cpp:165 ESP32

Restarts BLE advertising after a disconnect — but defers via bleRestartAdvertisingPending if the server isn't ready or an e-paper refresh is in progress, and no-ops if a client is already connected. On success it restarts advertising and refreshes the manufacturer data. The loop calls it both inside the post-wake window and in normal servicing to drain that pending flag.

ble_nrf_advertising_tick() ble_init.cpp:67 nRF

The nRF counterpart to the ESP32 advertising housekeeping. After a button press temporarily boosts the advertising interval for faster discovery, this tick reverts to the normal interval once the boost window expires, restarting advertising with the slower timing. Does nothing while the boost is still active.

imageWriteLogQuietFrame(data, len) display_service.cpp:1462 ESP32

A tiny logging predicate: true when the frame is an image-write chunk (opcode 0x0071) that's past the first chunk and should be logged quietly. The loop uses it purely to suppress per-frame serial spam during an image upload — the 5% progress meter reports instead.

03

WiFi / LAN session

3 functions
handleWiFiServer() wifi_service.cpp:170 ESP32

The LAN TCP server pump, run every pass. Detects the WiFi-up edge and starts the server; accepts an incoming client (replacing any previous one), reads available bytes into a receive buffer, and de-frames the stream (2-byte length prefix) — handing each complete frame to imageDataWritten(). Tears the session down on WiFi loss, buffer overflow, or an invalid frame length.

disconnectWiFiServer() wifi_service.cpp:160 ESP32

Closes the active LAN client: clears the encryption session, stops the socket, resets the server-connected flag and receive-buffer position. The loop calls it when the 10-second WiFi check sees the connection has dropped.

restartWiFiLanAfterReconnect() wifi_service.cpp:249 ESP32

Recovery path after WiFi comes back: closes any stale client, re-begin()s the TCP server on its port, and restarts the LAN service. Invoked from the loop's reconnect branch so the server is listening again after an outage.

04

Display & write-state watchdogs

2 functions
checkPartialWriteTimeout() display_service.cpp:180 Both

Watchdog for a stalled partial-write transfer. If one has been active longer than 15 minutes (900 000 ms), it logs an error and calls cleanup_partial_write_state() so a client that vanished mid-transfer can't wedge the device permanently.

cleanupDirectWriteState(bool refreshDisplay) display_service.cpp:1527 ESP32

Resets the entire direct-write upload state machine (all the directWrite* counters and flags back to zero). When refreshDisplay is set and the panel is powered, it also puts the panel to sleep (bb_epaper or Seeed path). The loop invokes it via the 15-minute direct-write timeout guard to recover stuck uploads.

05

Input, sensors & housekeeping

5 functions
processButtonEvents() device_control.cpp:411 Both

Polls power buttons (and, on ESP32, configured power-off buttons), then handles any pending button-ISR event: reads the pin, resolves logical press state (honoring inversion), packs button_id / press_count / state into the dynamic return data, and refreshes the advertising payload via updatemsdata(). On nRF it also boosts advertising so a press is discoverable quickly.

processTouchInput() touch_input.cpp:575 Both

Services GT911 touch controllers, rate-limited to a minimum interval and — on ESP32 — skipped entirely while a direct write or e-paper refresh is in progress (to keep the SPI/I2C buses clear). Supports both IRQ-edge and timed polling, with an I2C-failure backoff. No-ops when no touch controller is configured.

updatemsdata() display_service.cpp:1283 Both

Rebuilds the 16-byte BLE manufacturer-data advertising payload. Polls SHT40 / BQ27220 sensors, reads battery voltage and chip temperature, encodes them alongside the dynamic button data and a status byte (reboot flag, connection-requested, loop counter). The loop runs it on a 60-second cadence, and it's also triggered on button/sensor changes so the advertised state stays fresh.

processLedFlash() device_control.cpp:350 Both

The first call in every loop iteration. Advances a non-blocking LED flash animation by one step — but only if a flash is active and any inter-step delay has elapsed. Returns immediately otherwise, so it's essentially free when no LED sequence is running.

writeSerial(String, bool newLine = true) main.cpp:562 Both

The logging helper used throughout the loop. Writes to the dedicated log UART when OPENDISPLAY_LOG_UART is configured, otherwise to USB serial — and compiles to nothing when serial is disabled. Not control flow, but it's the most-called name in the function.
